# PTCGP DB — Design Document

This document defines the requirements and design decisions for PTCGP DB. It is the primary
reference for building the app. When a requirement is unclear or a significant design decision
is not covered here, stop and ask the user rather than making assumptions.

---

## Overview

PTCGP DB is a GUI application for tracking Pokémon TCG Pocket (PTCGP) card collections and
browsing the full database of cards, packs, and sets. It is primarily a serverless web application
— deployable from a static file host with no backend — and also packaged as a desktop app
(Windows and Linux) and mobile app (Android and iOS) from the same codebase. The app is fully
open source, free, and has no advertising. The primary consumer is the user who commissioned it,
but it will be made publicly available via GitHub.

---

## Technology Stack

- **Language**: Rust
- **UI Framework**: Dioxus — provides a unified compile target for web (WASM), desktop, and mobile
  from a single codebase
- **Styling**: Tailwind CSS exclusively. No custom CSS beyond Tailwind theming configuration.

## Workspace Structure

The project is a Cargo workspace containing three crates:

```
ptcgp-db/               ← workspace root
├── Cargo.toml          ← workspace manifest
├── ptcgp-db/           ← main app crate
│   └── src/            ← Dioxus UI, pages, components, routing
├── ptcgp-db-core/      ← shared business logic crate
│   └── src/            ← probability calculations, collection model, storage trait and backends
└── ptcgp-db-data/      ← generated data crate
    ├── codegen/        ← build script (entrypoint: codegen/main.rs); multiple files
    │                     reads ptcgp-data JSON, emits Rust const/static definitions to OUT_DIR
    ├── src/            ← hand-written wrappers around the generated code
    ├── ptcgp-data/     ← git submodule: card/set/pack/pull-rate data (JSON)
    └── ptcgp-images/   ← git submodule: all image assets
```

**`ptcgp-db-data`** contains all card, pack, set, and pull rate data as Rust `const`/`static`
structures generated at build time. The build script lives in `codegen/` (entrypoint:
`codegen/main.rs`) and writes generated code to Cargo's `OUT_DIR`. The `src/` directory contains
hand-written wrappers that make the generated code convenient to use. The crate has no runtime
dependencies on JSON parsing or databases. Initial compilation may take up to 10 minutes;
subsequent builds are incremental.

**Ordering guarantees on data sequences**: all sequences exposed by this crate are pre-sorted.
Sequences with a canonical display order (e.g., `CardVersion::ALL`) are already in that order as
defined by the design doc. Sequences without a special display order are sorted for binary search:
alphabetically for string-keyed data, numerically by ID for ID-keyed data. Callers can rely on
these orderings and use binary search for lookups rather than sorting at runtime.

**`ptcgp-db-core`** contains everything that is pure logic with no UI dependency: the probability
calculation engine, the collection model (owned counts, profile aggregation), the storage trait
and its per-platform implementations (IndexedDB for web, file-based for desktop/mobile), data
format versioning and migration, and any other types shared between the data and app crates.

**`ptcgp-db`** is the Dioxus application: all pages, components, routing, and UI-specific logic.
It depends on both `ptcgp-db-core` and `ptcgp-db-data` but contains no business logic itself —
that belongs in `ptcgp-db-core`.

## Data Sources

All card data and image assets are provided by two companion repositories.

### ptcgp-data

Git submodule at `ptcgp-db-data/ptcgp-data`. Contains all card, pack, set, and pull rate data in two formats: JSON (checked into the repo,
always available) and SQLite (generated by running `cargo run --release --bin db`, not checked
in). **Use the JSON format** as the source for code generation — it is always present and does
not require a build step to produce.

All JSON lives under `data/`. Key files:

| Path | Contents |
|---|---|
| `data/sets.json` | All sets (code, name, series, `availability.start`, `availability.end`, `is_promo`, card count); `availability` is null for promo sets |
| `data/rarities.json` | All rarity tiers (code, name, group, symbol count, craft/dupe costs) |
| `data/elements.json` | Energy types (symbol letter, name) |
| `data/base_pokemon.json` | Pokédex reference (natdex number, name) |
| `data/card_sources.json` | How cards with non-pack sources can be obtained (code, description) |
| `data/pack_variant_names.json` | Display names for variant codes (`normal`, `rare`, `plus1`, `themed`) |
| `data/sets/{SET}/set.json` | Full set detail including pack subtitle list |
| `data/sets/{SET}/cards/{NUM:03}.json` | One card version per file |
| `data/cards/{ID:05}.json` | One abstract card per file |
| `data/pull_rates/{SET}/{subtitle}.json` | Pull rate data for one pack (subtitle is lowercase with spaces replaced by `_`, e.g., `vol_1.json`) |

### ptcgp-images

Git submodule at `ptcgp-db-data/ptcgp-images`. Contains all image assets. Key directories:

| Path | Contents |
|---|---|
| `cards/` | Card images, one per card version (~300 KB each, 3,289 total, >1 GB combined) |
| `cards/back.png` | Special image of the back of a card if needed. All cards have the same back image. |
| `sets/logos/`, `sets/icons/` | Set logos and icons |
| `packs/art/`, `packs/logos/` | Pack artwork and logos |
| `rarities/symbols/`, `rarities/icons/` | Rarity symbols (inline text) and icons |
| `elements/symbols/`, `elements/icons/` | Element symbols (inline text) and icons |
| `elements/icons/no_cost.png` | Special icon for 0-energy attack cost |
| `card_sources/` | Icons for non-Pack card acquisition methods |

All images must be referenced using Dioxus's `asset!()` macro, never by raw string paths. The
`asset!()` macro ensures correct path resolution and bundling across all platforms.

### Data Consumption Strategy

- **Non-image data**: Consumed at build time via the `codegen/` build script in `ptcgp-db-data`,
  which reads the `ptcgp-data` JSON files and emits `const`/`static` Rust data structures to
  `OUT_DIR`. This eliminates all runtime overhead from JSON parsing or database queries, and
  allows the compiler to optimize data access. The generated code lives in the `ptcgp-db-data`
  crate so that incremental builds skip it when unchanged. Initial compilation may take up to 10
  minutes; this is acceptable.

- **Images**: Bundled at build time using `asset!()` via `dx`. Images are loaded lazily by the
  UI — only images for currently visible components are loaded.

### Image Path Conventions

All image paths in `ptcgp-images` follow consistent derivable conventions. The `asset!()` macro
call should use these paths relative to the `ptcgp-images` root.

| Image | Path pattern | Notes |
|---|---|---|
| Card image | `cards/{SET}/{NUM:03}.png` | e.g. `cards/A1/025.png` |
| Set logo | `sets/logos/{SET}.png` | e.g. `sets/logos/A2b.png` |
| Set icon | `sets/icons/{SET}.png` | |
| Pack art | `packs/art/{SET}/{subtitle}.png` | subtitle is lowercase, spaces → `_` |
| Pack logo | `packs/logos/{SET}/{subtitle}.png` | same subtitle convention |
| Rarity icon | `rarities/icons/{group}/{count}.png` | group = `diamond`/`star`/`shiny`/`crown` (lowercase); count = `RarityClass::count()` |
| Rarity symbol | `rarities/symbols/{group}/{count}.png` | same as rarity icon |
| Element icon | `elements/icons/{element}.png` | element name lowercased, e.g. `grass.png` |
| Element symbol | `elements/symbols/{element}.png` | same element convention |
| No-cost icon | `elements/icons/no_cost.png` | used for 0-energy attack cost |
| Card source icon | `card_sources/{code}.png` | `CardSource::name()`, lowercased, spaces → `_`; used for non-Pack card sources |

All derivations from data fields are done at compile time in `codegen/` or via constants — never
construct image paths from runtime strings.

## Domain Concepts

Understanding the PTCGP data model is essential for implementing the app correctly.

### Abstract Cards vs. Card Versions

The data distinguishes two levels of card identity:

- **Abstract card** (`data/cards/{ID:05}.json`): The game-mechanical identity — a Pokémon or
  Trainer with a specific name, stats, attacks, and effects. One abstract card may be printed
  as many different physical cards across different sets.

- **Card version** (`data/sets/{SET}/cards/{NUM:03}.json`): One specific physical card — a
  particular set, collector number, rarity, illustrator, and finish (foil or not). This is what
  users actually own copies of.

Key linking fields (expressed as `ptcgp-db-data` API calls):
- `CardVersion::card()` → the abstract card for a given version
- `Card::versions()` → all card versions that share an abstract card's mechanics
- `CardVersion::duplicates()` → other card versions with the same rarity, illustrator, and
  finish (the same physical card released again in a different set)
- `CardVersion::is_original()` → `true` for the original printing; `false` for a reprint

**Owned counts are always tracked per card version**, not per abstract card. When displaying
"how many do I own," the count refers to that specific card version.

### Sets and Series

Sets are identified by a short code (e.g., `A1`, `B2a`, `P-A`). Each set belongs to a series
identified by a single letter (e.g., `A`, `B`; new series are added as the game expands).
`Set::is_promo()` returns `true` for promo sets (see Promo Cards below).

### Set Availability

Non-promo sets have two availability dates:

- **`Set::release_date()`**: the date the set became available
- **`Set::retirement_date()`**: the date the set became unobtainable, or `None` if still obtainable

A set is considered **unobtainable** if `Set::retirement_date()` returns a date in the past.
Currently the only unobtainable set is **A4b** — its packs can no longer be opened.

Promo sets return `None` from both `Set::release_date()` and `Set::retirement_date()` — they
have no availability window (individual promo cards may be limited-time, but that is not tracked
at the set level).

Unobtainable sets remain in the catalog and count toward completion by default. However,
unobtainable packs are excluded from "next pack to open" recommendations everywhere in the app
by default (see §Summary Page). A global setting can hide unobtainable sets
from the entire app; see Settings.

### Packs, Variants, and Slots

Sets have one or more packs identified by a subtitle (e.g., "Charizard", "Mewtwo"). Pull rate
data lives at `data/pull_rates/{SET}/{subtitle}.json`. Promo sets also have packs, but they
work differently — see Promo Cards.

**Pack display name**: every set has a name (`Set::name()`, e.g., "Genetic Apex") and every pack
has a subtitle (`Pack::subtitle()`, e.g., "Charizard"). Many sets have only one pack, in which
case the subtitle is typically identical to the set name. Use `Pack::title()` to get the correct
full display name — it implements this logic:

- If `Set::name() == Pack::subtitle()`: display only the set name (no repetition)
- Otherwise: display `{set.name}: {pack.subtitle}` (e.g., "Genetic Apex: Charizard")

Each pack has **variants** that determine what cards may appear and with what probability:

| Variant | Approximate Rate | Slot Count |
|---|---|---|
| Regular Pack | ~94–99.95% | 5 |
| Regular Pack +1 | ~5–8% | 6 (bonus slot) |
| Rare Pack | 0.05% | 5 (star-rarity cards) |
| Themed Rare Pack | 0.005% | 5 (featured cards) |

Not all packs have all variants. Each variant's exact pull rate is `PackVariant::pull_rate()`
(a `Prob`). Variant rates for a given pack sum to 1.

Each variant has **slots** (one per card yielded), accessible via `PackVariant::slots()`. Slot 0
is the first card shown to the player, slot 1 the second, and so on (`PackSlot::pull_number()`).
Pull rates differ per slot.

For each slot, the data provides:
- **Rarity rates** (`PackSlot::rarities()`, a `&[RarityPullRate]`): probability of each rarity
  appearing, split into non-foil (`RarityPullRate::normal_pull_rate()`) and foil
  (`RarityPullRate::foil_pull_rate()`), each a `Prob`
- **Card rates** (`PackSlot::card_versions()`, a `&[CardVersionPullRate]`): per-card probability
  for each slot (`CardVersionPullRate::pull_rate()`); cards absent from the list have an implicit
  rate of `Prob::ZERO`

All rates are `Prob` values — exact rational arithmetic. Intermediate probability calculations
must use `Prob`; only convert to a float (`Prob::as_f64()`) or percentage for final display.

### Promo Cards

Promo cards belong to promo sets (`Set::is_promo()` returns `true`). Normal sets never contain
promo cards.

Promo sets have packs with pull rate data like normal packs, but structured differently: each
promo pack has a single slot, a single variant (`normal`) with rate 1 (100%), and typically
around 5 possible cards.

**Promo packs are excluded from all probability calculations in the app** — they do not appear in
"next pack to open" recommendations or Summary page query results. Rationale: promo cards are
limited-time and unobtainable once their event ends; users do not make pack-opening decisions
based on promo pull rates.

Some cards are not available from any pack at all; they are obtained through other in-game means
(events, missions, shop). See Card Sources below.

Promo cards are tracked in the collection and count toward set completion like any other card.
Where set completion is shown, promo sets are included but without any "chance to pull" data.

### Card Sources

Every card version has exactly one source (`CardVersion::source()`, a `&CardSource`) indicating
how it is obtained in PTCGP. The special source whose `CardSource::name()` is `"Pack"` means the
card is available from packs and has associated pull rate data and assigned packs
(`CardVersion::packs()`). All other sources mean the card is obtainable only through other
in-game means (missions, events, shop, etc.), and these cards have **no pull rate data and no
assigned packs** (`CardVersion::packs()` is empty) — even if they belong to a non-promo set.

The only known non-promo card with a non-Pack source is **A1 283** (IM Mew), obtainable only by
completing a mission to collect one of each Kanto Pokémon.

When displaying card details for a card with a non-Pack source, show the source description so the
user knows how to obtain it.

### Rarities

Rarities are organized in a three-level hierarchy:

- **Specific rarity** — a unique code (e.g., `C`, `AR`, `SR`, `IM`) with a human-readable name
  (e.g., "Common", "Art Rare", "Super Rare", "Immersive Rare"). Defined in `data/rarities.json`.
- **Rarity group** — the symbol shape used: Diamond, Star, Shiny, or Crown.
- **Rarity class** — the 2-tuple of (group, symbol count), e.g., Diamond-2 or Star-2. This is
  the granularity at which rarity icon and symbol images are defined. Star-2 is the only rarity
  class that contains more than one specific rarity (SR and SAR).

**UI convention**: rarity is always represented in the UI by its **rarity class icon or symbol
image** — the icon for Diamond-2 is literally an image of two diamond shapes; the symbol is a
text-height version of the same. Do not display rarity codes (`C`, `AR`, etc.) or group names
("Diamond", "Star") as user-facing text. Specific rarity names (e.g., "Common", "Super Rare")
are only shown as supplemental detail, specifically in the Card Details view.

### Card Codes

A card is identified in the UI by its **card code**: the set code followed by the card's
zero-padded number within the set, e.g., `A2b 025` or `B2 134`. The number field on a card
version is its collector number within the set.

## Probability Calculations

This section defines how pull probabilities are computed. These calculations power the Summary
page's "next pack to open" and query results.

**Promo packs are always excluded from these calculations.**

### Per-Card Pull Rate for a Pack

For a given pack and a target card, the probability of obtaining that card from a single opening
is:

```
P(card appears in at least one slot) =
  sum over all variants v:
    v.pull_rate() × (1 - product over all slots s in v.slots():
      (1 - card_rate(card, s)))
```

Where `card_rate(card, s)` is the `Prob` from the `CardVersionPullRate` entry for `card` in
`s.card_versions()`, or `Prob::ZERO` if the card is absent. The inner product is P(card misses
every slot | variant v); subtracting from 1 gives P(card in at least one slot | variant v).
Slots within a variant are independent draws. All arithmetic uses `Prob` (exact rational
arithmetic). Convert to a percentage only at final display time.

### Probability of Pulling Any "Desired" Card

For features like "next pack to open," the target is a set of desired cards (e.g., all unowned
cards). The probability of obtaining at least one desired card from a single opening is:

```
P(at least one desired card) =
  sum over all variants v:
    v.pull_rate() × (1 - product over all slots s in v.slots():
      (1 - sum over all desired cards c in s.card_versions():
        c.pull_rate()))
```

Within a single slot, cards are mutually exclusive (only one card appears per slot), so the
inner sum is P(any desired card drawn in slot s). The product accumulates P(no desired card in
any slot | variant v); subtracting from 1 gives a true probability in [0, 1].

When the **Merge duplicate printings** setting is enabled, treat all versions within a duplicate
group as a single card for the purpose of determining desired status. The combined count (sum
across all versions in the group) is used as the owned count for that card. If multiple versions
from the same group can appear in the same pack, their individual pull rates still sum normally —
the merging only affects whether the card is considered desired, not how pull rates are aggregated.

### Completion Formula

The completion percentage for any query is:

- **Target** (T): the ownership count being worked toward (e.g., 1 for "own at least one", 2
  for "own at least two for deck building")
- **Numerator**: `sum of min(count(c), T)` for all card versions `c` matching the query
- **Denominator**: `(number of card versions matching the query) × T`

Where `count(c)` is the summed owned count across all active profiles (see Profiles).

The Summary page set completion is the special case: T = 1, all card versions in the set.

When the **Merge duplicate printings** setting is enabled, the formula operates on physically
unique cards rather than individual card versions. Each duplicate group is collapsed into one
entry: the count used is the sum across all versions in the group, and the group counts as one
card in the denominator regardless of how many versions fall within scope.

## User Data and Persistence

### What Is Stored

The app stores the following user data locally:
- **Collection data**: owned count per card version, per profile
- **Profile metadata**: profile names, primary profile designation
- **Saved queries**: named Summary page filter configurations (not profile-specific)
- **App settings**: active profile(s), and all AppSettings preferences (dark/light mode, ignore
  unobtainable sets, ignore Premium Mission cards, ignore Gold Shop cards, merge duplicate printings)

### Storage Backends

Define a storage trait and implement it per platform:

- **Web**: IndexedDB via `web-sys` (or a lightweight wrapper such as `rexie`). IndexedDB is
  required over LocalStorage because it is asynchronous (non-blocking) and more robust for
  structured data.
- **Desktop / Mobile**: A JSON file in the platform-appropriate user data directory, located
  using the `dirs` crate:
  - Linux: `~/.local/share/ptcgp-db/`
  - Windows: `%LOCALAPPDATA%\ptcgp-db\`
  - macOS: `~/Library/Application Support/ptcgp-db/`
  - Android/iOS: platform data directory provided by `dirs`

### Auto-Save

All changes to collection data are saved automatically, debounced 2 seconds after the last
change. The app loads the previous session's data on startup automatically.

### Data Format and Versioning

Both internally managed save data and user-exported data must include a format version
identifier. As the app evolves, the save format may change. When loading data with an older
version identifier, the app must automatically migrate it to the current format. Migration must
be lossless where possible.

The internal save format may use whatever representation is best for the storage backend. The
export format (for user-facing import/export) must be JSON.

## Profiles

A profile is a named collection of card ownership data. Users can create and delete profiles
freely. Each profile has a unique name. One profile is designated the **primary profile**, which
serves as the default in any context where a single profile must be chosen automatically. On
startup, the app opens with the primary profile active in single-profile mode. Future features
that require a default profile selection should also use the primary profile.

### Multi-Profile Aggregation

Multiple profiles can be active simultaneously on most pages. When multiple profiles are active:

- **Owned count** for a card version = sum of that card version's count across all active profiles
- A card version is considered **owned** if and only if this sum is > 0
- This aggregated count is used consistently for: displayed counts, completion percentages,
  probability calculations, and all saved queries

This model applies throughout the app. The Trade page follows the same aggregation rule for
active profiles (the destination), but additionally surfaces inactive profiles as individual
sources (cards to give). See the Trade Page section.

### Editing Counts with Multiple Profiles Active

Editing owned counts is only possible when exactly one profile is active. When multiple profiles
are active, count editors (spinners, increment/decrement buttons) are disabled and the displayed
count is the aggregate sum. The user must switch to a single active profile to make edits.

## State Management

Global app state is managed using Dioxus context providers. State is provided at the root of the
component tree using `use_context_provider` and consumed anywhere in the tree using `use_context`.
Do not use global statics or thread-locals for app state — context providers integrate with
Dioxus's reactivity system and are the idiomatic approach.

The following contexts must be provided at the app root:

- **`ProfileStore`** — owns all profiles and their collection data, tracks which profiles are
  currently active, exposes methods to read/write owned counts, create/rename/delete profiles,
  and change the primary profile. Also owns the storage backend instance and triggers auto-saves.
- **`AppSettings`** — dark/light mode preference and any other app-wide settings. Persisted via
  the storage backend.
- **`SavedQueries`** — the list of saved Summary page filter configurations. Persisted via the
  storage backend.

Components read from these contexts reactively — any component that reads a context value will
automatically re-render when that value changes. Write operations go through methods on the
context types (not direct field mutation) so that saves are triggered consistently.

Business logic (probability calculations, collection aggregation, migration) lives in
`ptcgp-db-core` and is called by the context types, not by UI components directly.

## UI Guidelines

### Image Usage

Use images from `ptcgp-images` liberally, following these conventions:

- **Inline text contexts** (e.g., within effect descriptions): use `rarities/symbols/` and
  `elements/symbols/` images to replace text references. The data strings in `ptcgp-data` contain
  element placeholders like `[R]` (Fire) and `[G]` (Grass) that are designed to be replaced with
  element symbol images.
- **Non-text contexts** (e.g., filter chips, icons): use `elements/icons/` and `rarities/icons/`.
- **Sets**: use `sets/logos/` when space allows, `sets/icons/` when the logo is too large.
- **Packs**: use `packs/art/` for full artwork, `packs/logos/` for smaller contexts.
- **Cards without a pack** (non-Pack source): use `card_sources/` images in place of pack logos.
- **Cards**: use `cards/` images whenever a card is referenced and space permits showing the full
  card. Card images average ~300 KB; load them lazily (only when visible).
- **0-energy attack cost**: use `elements/icons/no_cost.png`.

## App Structure and Navigation

The app consists of multiple pages with persistent navigation (e.g., a sidebar or nav bar). The
navigation is always visible and allows switching between pages.

### Responsive Design

Responsive layout using Tailwind's breakpoint system (`sm:`, `md:`, `lg:`, `xl:`) is a hard
requirement. Every page must render correctly and be fully usable at any viewport width, from a
narrow phone screen to a wide desktop window. Do not design only for one viewport size and treat
others as an afterthought.

Key layout adaptations expected at narrow viewports:

- **Navigation**: a persistent sidebar is appropriate at wide widths; at narrow widths it should
  collapse to a bottom navigation bar, a hamburger menu, or equivalent.
- **Card Catalog + Detail panel**: the side-by-side layout (list left, detail panel right) is for
  wide viewports only. At narrow widths, the list fills the full width and tapping a card
  navigates to the Card Details view as a full-screen page.
- **Filter toolbars**: at narrow widths, filter controls may need to collapse behind a toggle
  button (e.g., a "Filters" button that reveals a drawer or modal) rather than occupying
  persistent horizontal space.
- **Profile selector**: must remain accessible at all viewport widths.

### Routing

The web version uses **hash routing** (e.g., `/#/summary`, `/#/catalog`). Hash routing requires
no server-side configuration and works on any static file host. Dioxus's router supports hash
routing natively. Do not use history-API routing — it requires server-side redirect configuration
that a static host cannot provide.

The desktop and mobile versions use Dioxus's native router without hash routing.

### Default Page

The **Summary page** is the default. It is the first page shown when the app opens.

### Pages

- Summary (default/home; includes Analysis query features)
- Card Catalog
- Trade
- Profile Manager
- Import / Export
- Settings

Card Details is not a standalone page — see the Card Catalog section.

There is no separate Analysis page. The Summary page includes the full filter toolbar, goal input, saved queries, and completion calculations originally specified for Analysis.

### Profile Selector

A profile selector is permanently visible in the navigation bar. It shows the currently active
profile(s) and allows the user to switch profiles from any page without navigating away.

The selector supports multi-select — the user can activate more than one profile simultaneously
to aggregate data across them. Single-profile selection is by far the common case; the UI should
make selecting one profile the path of least resistance, with multi-select as an explicit
secondary action (e.g., a dropdown or popover that lists all profiles with checkboxes).

Profile-agnostic pages (Settings, Import/Export) may hide or disable the selector since the
active profile does not affect their content.

## Summary Page

The Summary page is the default home screen. It always supports multiple active profiles. It
combines collection overview (set completion, best pack to open) with the query features
originally planned for a separate Analysis page — no separate Analysis page exists.

### Filter Toolbar

The Summary page includes the shared filter toolbar in Summary mode. It supports all filter
dimensions (set, pack, series, rarity, element, stage, ex, mega, foil, card source, card kind,
obtainable) except name search, which is omitted because the summary view shows set/pack rows
rather than individual cards.

Additional controls specific to Summary mode:

- **Goal number input** (T, default 1): cards where `count < T` are "desired" and drive pack
  probability calculations; all matching cards count toward the completion denominator. This
  replaces the Card Catalog's owned-count threshold filter.
- **Any version owned** toggle: when on, a card version is treated as owned if the aggregate
  count of *any* version of the same abstract card is > 0.
- **Obtainable** defaults to "obtainable only": unlike the Card Catalog where this defaults to
  unset, Summary defaults to restricting the card pool to cards from obtainable sets. Hidden when
  the global "Ignore unobtainable sets" setting is on.

### Saved Queries

Users can save the current filter configuration under a user-defined name and reload it from a
dropdown. Saved queries are stored as part of local app data and are **not** profile-specific.
The query dropdown trigger displays the name of the currently loaded query and shows an asterisk
when the active filter has been modified since the query was loaded.

### Content

**Next pack to open**
The non-promo pack with the highest probability of yielding a desired card (aggregate count < T)
from the filtered card pool. Display the pack name and the probability as a percentage. If no
desired cards exist (all matching cards satisfy T), display a collection-complete message.
Clicking a pack navigates to the Card Catalog with the active Summary filters applied.

Unobtainable packs are excluded by default (the Obtainable filter defaults to "obtainable
only"). When the "Ignore unobtainable sets" setting is off, the Obtainable filter may be cleared
to include unobtainable packs. When the setting is on, the filter is hidden and unobtainable
packs are always excluded.

**Set completion**
For each set matching the active filters:
- Completion % using the completion formula with goal T and the filtered card pool
- Best pack probability for desired cards in that set (expandable per-pack breakdown)
- Promo sets: completion % only, no probability data
- Sets and packs are identified by their logo and icon images
- Clicking a set navigates to the Catalog with active Summary filters applied; the clicked set
  overrides any active set filter (packs cleared). Clicking a pack overrides any active pack
  filter (sets cleared). Goal, any-version, owned-count, and name filters are reset in the
  Catalog regardless.

**Overall totals**
Completion % across all matching sets, with owned/total counts and a progress bar.

**Collection history graphs** (nice-to-have)
Line graphs showing collection completion % over time — one overall and one per set. Requires
storing a timestamp whenever a card count changes. Implement only after core features are working.

## Card Catalog Page

A searchable, filterable, sortable virtual list of all card versions. This is one of the primary
pages of the app.

### Multi-Profile Behavior

Supports multiple active profiles. When multiple profiles are active, displayed counts are the
aggregate sum and count editors are disabled.

### Virtual List

The list **must** be a true virtual list — only rows currently visible in the viewport plus a
small scroll buffer are rendered in the DOM. This is a hard requirement:

- There are **3,289** total card versions as of writing, expected to grow ~1,000–1,500 per year
- Card images average ~300 KB each, totalling over 1 GB
- Off-screen rows must be removed from the DOM entirely, not hidden — unloaded images must not
  consume memory

The list updates immediately on every filter change, including on every keystroke in the text
search. The card filtering engine must be designed for performance. Baking the data into static
Rust structures (the code generation strategy) is critical for this.

**Implementation**: use the virtual list component from `dioxus-primitives`, which is already an
approved dependency. Row layout should use styled `div` elements acting as columns rather than
an HTML `<table>` — true table elements interact poorly with DOM virtualization (sticky headers,
fixed column widths, and border-collapse all create complexity). If the `dioxus-primitives`
component proves insufficient for the catalog's needs, discuss a custom implementation with the
user before proceeding. Do not use any other virtual list crate.

### Each Row

At minimum, each row includes:
- Card image (thumbnail size)
- Card code (e.g., `A2b 025`)
- Card name
- Rarity class icon
- Pull rate: the highest aggregate pull rate for this card across all non-promo packs, displayed
  as a percentage. Cards with no non-promo packs (promo cards, non-Pack-source cards) show N/A.
  Hovering over the value shows tooltip text identifying the source pack, using the pack display
  name convention defined in Packs, Variants, and Slots.
- Owned count with an increment/decrement spinner (disabled when multiple profiles are active)

Any field displayed in the list that has a meaningful sort order should be sortable. Any
sortable dimension should be displayed in the list.

### Row Visual Styling

Rows use a layered visual styling system. Effects are additive — a foil Shiny card gets both its
Shiny and foil effects simultaneously.

**Layer 1 — Element tint (base background)**
Pokémon card rows have a very subtle background tint (approximately 10–15% opacity) in their
element's theme color. Trainer card rows have no tint (use the default theme background). The
element color mapping must be defined once as custom Tailwind theme colors (see UI Guidelines)
and referenced consistently. The tint must remain legible in both light and dark mode.

**Layer 2 — Rarity group effect (border/overlay)**
Applied based on the card's rarity group (rarity codes listed for reference only — not shown to users):

| Rarity Group | Rarities (internal) | Effect |
|---|---|---|
| Diamond | C, U, R, RR | No special effect |
| Star (Full Art) | AR, SR, IM, SAR | Subtle gradient border in the card's element color at higher saturation |
| Shiny | S, SSR | Static sparkle/glitter overlay or pattern on the row |
| Crown | UR | Gold border or gold-accented row styling |

**Layer 3 — Foil effect**
Foil card versions (`CardVersion::is_foil()` is `true`) get a static rainbow/iridescent CSS
gradient applied to the row border or background, regardless of rarity group.

**Layer 4 — Premium source tint**
Cards whose `CardVersion::source().name()` is `"Premium Mission"` or `"Gold Shop"` require
spending real money to obtain.
These rows get a distinct gold/amber tint or border overlay to signal this — separate from the
element tint — so users can immediately recognize that obtaining the card has a monetary cost.
The exact color should feel "premium" (gold or amber), remain legible in both light and dark
mode, and be visually distinct from both the element tint and rarity effects.

All effects must be **static CSS** (no animations). Animated effects in a virtual list with
many simultaneously visible rows create visual noise and can degrade scroll performance.

### Sorting

Default sort: **canonical set order, then collector number within the set.**

Canonical set order is defined as:
1. Series alphabetically (A, B, C, …)
2. Within a series: by `Set::release_date()` ascending; promo sets (`Set::is_promo()` is `true`,
   and `Set::release_date()` returns `None`) always sort last within their series regardless of
   when promo cards were released
3. Within a set: collector number ascending (numeric)

Example order: A1, A1a, A2, …, A4b, P-A, B1, B1a, …, P-B

Do not sort sets alphabetically by code — that ordering breaks if the game ever reaches a series
whose letter matches the promo set prefix format.

Additional sort dimensions:
- **Name** — alphabetical
- **Owned count** — numeric
- **Rarity** — sorted by rarity class: groups ordered Diamond < Star < Shiny < Crown; within a
  group, by symbol count ascending (e.g., ◆ < ◆◆ < ◆◆◆ < ◆◆◆◆)
- **Element** — canonical order: Grass, Fire, Water, Lightning, Fighting, Psychic, Darkness,
  Metal, Dragon, Colorless
- **Pull rate** — the highest aggregate pull rate for this card across all non-promo packs (same
  value shown in the row); cards with no non-promo packs sort last

### Filters

All filters default to unset (no effect). When all are unset, all card versions are shown. The
user can reset any individual filter to unset. When filters produce zero results, show an
explicit "no results" message.

| Filter | Behavior |
|---|---|
| **Name / Number** | Text input; case-insensitive substring match on card name. Also matches collector number (e.g., "42" matches all cards numbered 42 in their set). |
| **Series** | Single-select. Filter to one series (e.g., A or B). |
| **Set** | Multi-select. Filter to one or more sets (e.g., A2b, B2). |
| **Pack** | Multi-select. Filter to cards available in one or more packs. |
| **Rarity** | Multi-select. Filter to one or more rarity classes, displayed as their class icon images (e.g., ◆◆ for Diamond-2, ★ for Star-1). |
| **Card Kind** | Single-select: Pokémon or Trainer. |
| **Ex** | Toggle: show only ex Pokémon, or exclude ex Pokémon. |
| **Mega** | Toggle: show only Mega ex Pokémon, or exclude them. |
| **Stage** | Single-select: Basic, Stage 1, Stage 2. |
| **Element** | Multi-select. Filter Pokémon by element. |
| **Foil** | Toggle: show only foil card versions, or exclude foil card versions (`CardVersion::is_foil()`). |
| **Card Source** | Multi-select. Filter by `CardVersion::source()` (e.g., `"Pack"`, `"Premium Mission"`, `"Gold Shop"`). |
| **Obtainable** | Toggle: when set, restrict to cards from obtainable sets only, or to cards from unobtainable sets only. Hidden when the global "Ignore unobtainable sets" setting is on (unobtainable sets are already excluded). |
| **Owned count** | Threshold: e.g., `= 0`, `< 2`, `>= 3`. Operates on the aggregate count across active profiles. |

## Trade Page

The Trade page assists users who play PTCGP on multiple accounts — each represented by a profile
— in deciding which cards to share or trade between accounts and in recording completed transfers.

**Active profiles are the destination** (the profiles that should receive cards); inactive
profiles are the sources (alternates). Active profiles follow the same aggregation model as the
rest of the app — their counts are summed into a single aggregate that represents the destination.
Inactive profiles are each evaluated individually as potential sources.

Recommended Shares and Recommended Trades require at least one inactive profile as a source.
When all profiles are active or only one profile exists, those sections show an empty state — in
the single-profile case, a prompt to create an additional profile; in the all-active case, a note
to deselect at least one profile to use it as a source. The Trade Candidates section is always
shown regardless.

### Collection Goal

The page exposes the same filter toolbar and goal number input (T) as the Summary page,
defining which cards the destination "wants." Default: T = 1 (own at least one of each card).
Adjusting the goal or filters recalculates all recommendations instantly. A card is a candidate
to receive if the aggregate count across all active profiles is below target.

### Card Eligibility

PTCGP restricts which cards can move between accounts:

- **`CardVersion::is_tradable()`** — must be `true` for a card to be eligible for shares or
  trades; cards where it is `false` are excluded from all recommendations
- **Shareable** — additionally requires the card's rarity group to be Diamond (C, U, R, or RR)
- **Tradeable** — only `CardVersion::is_tradable()` is required; any rarity class is eligible

The global "Ignore" settings (unobtainable sets, Premium Mission, Gold Shop) apply to the Trade
page as well — cards excluded by those settings do not appear in any recommendations.

When **Merge duplicate printings** is enabled, the merged count (sum across all versions in a
duplicate group) is used when determining need and excess. When a recommendation names a specific
card version to trade or share, always use the version where `CardVersion::is_original()` is
`true` — never a reprint — to keep recommendations consistent.

### Recommended Shares

A **share** is a once-per-day, one-way transfer of a Diamond card from one account to another,
free of charge. Recommendations identify the highest-value shares to perform:

- **Candidate cards**: `CardVersion::is_tradable()` is `true`, Diamond rarity group, and the
  aggregate count across active profiles is below target for that card
- **Source**: any inactive profile with count > 0 for that card — source profiles are treated as
  a pool to drain into the destination, so any owned copy is a candidate regardless of how many
  the source would have left
- **Ranking**: by the receive-value formula below, highest first. When multiple source profiles
  can supply the same card, prefer the one with the highest count (most copies to spare), and
  note the alternatives so the user knows they have flexibility.

**Receive-value formula** (used for both shares and trades):

```
needed = T - count   (aggregate copies still needed to reach target)
value  = 1.0 / (max_pull_rate × needed)
```

Higher value = higher priority for the destination to receive. Value rises as `max_pull_rate`
falls (harder to pull from packs) or as `needed` falls (closer to completing that card). Only
cards where `needed > 0` are candidates.

Cards where `max_pull_rate = 0` (non-Pack-source or promo cards the destination still needs)
are unobtainable from packs entirely — shares and trades are the only way to get them. These are
shown as a separate highest-priority tier above the ranked list rather than being excluded.

### Recommended Trades

A **trade** is a mutual exchange between two accounts where both cards must belong to the same
**rarity class** (same symbol type and symbol count). The trade costs a renewable in-game
currency; currency cost is not factored into recommendations.

For each (source profile, rarity class) pair, the best trade is selected independently:

- **Card B** (destination receives, source gives): among cards where `CardVersion::is_tradable()`
  is `true` in that rarity class, where the source count > 0 and the aggregate destination
  count < T, choose the one with the **highest receive-value** — the card the destination most
  benefits from getting.
- **Card A** (destination gives, source receives): among cards where `CardVersion::is_tradable()`
  is `true` in that same rarity class, where the aggregate destination count > T (excess > 0) and
  the source count < T (source wants it), choose the one with the **lowest trade-candidate value**
  — the card cheapest for the destination to part with (most surplus, most re-obtainable).

Both must exist for a recommendation to be generated for that (source, rarity class) pair. Rank
recommendations by the receive-value of Card B. Display both sides of each proposed trade clearly
— which card the destination gives and which it receives.

### Recording Transfers

When a share or trade is completed in the game, the user records it in the app as a **transfer
of ownership**: subtract 1 from the source profile's count and add 1 to the destination
profile's count for a given card version.

Each recommendation includes a **"Record transfer"** action that performs this update
automatically for that specific recommendation. For out-of-band transfers not covered by a
recommendation, users edit counts directly in the Card Catalog.

### Trade Candidates

A ranked list of cards from the destination (active profiles, aggregated) that are the best
candidates to give away in trades — useful for deciding what to offer when trading with other
players, and available even when only one profile exists.

**Eligibility**: only cards where:
- `excess > 0` where `excess = count - T` using the aggregate count across active profiles
- `max_pull_rate > 0` (the card appears in at least one non-promo pack); cards with no non-promo
  packs (promo cards, non-Pack-source cards) are omitted entirely since their re-obtainability
  cannot be expressed as a pull rate

**Value formula**:

```
value = 1.0 / (max_pull_rate × excess)
```

Where `max_pull_rate` is the card's highest aggregate pull rate across all non-promo packs (the
same value shown in the Card Catalog row) and `excess` is as defined above. Lower value = better
trade candidate. The list is sorted ascending by value.

Intuitively: value falls as either factor grows, so a card that is both easy to re-obtain and
held in large surplus scores the lowest and appears first.

**Unobtainable set cards**: cards from sets whose `Set::retirement_date()` is in the past cannot
be re-obtained from packs, making them more precious despite having a non-zero `max_pull_rate`
from historical data. These cards are **excluded from the list by default** and shown only when the
user opts in via a toggle (consistent with how unobtainable packs are handled elsewhere). When
shown, they are visually flagged (e.g., a warning indicator) to signal they cannot be re-obtained.

## Profile Manager

A view (page, modal, or collapsible panel — implementation's choice) for managing profiles.
Must support:

- Creating a new profile (with a unique name)
- Renaming a profile
- Deleting a profile — requires a confirmation dialog before proceeding
- Changing the primary profile

When a profile is deleted:
- If it was the **primary profile**, automatically promote another profile to primary. The new
  primary is whichever remaining profile has the largest total owned card count (sum of all card
  version counts). If multiple profiles tie, pick arbitrarily among them.
- If it was **currently active**, remove it from the active set. If it was the only active
  profile, activate the primary profile instead.

## First Run

When the app is opened for the first time and no save data exists, show a dedicated onboarding
screen (full page or modal) before presenting the main app. The onboarding screen must:

1. **Prompt the user to name their first profile.** The UI requires a name to submit the form.
   The named profile becomes the primary profile. (Dismissal without submitting is handled
   separately — see below.)

2. **Offer a prominent import option.** The first launch is the most natural time for a returning
   user (e.g., someone moving to a new device) to import existing data. The import action should
   be clearly visible on the onboarding screen — not buried. If the user imports data, the
   imported profiles replace the need to create a new one; the app should proceed directly after
   a successful import without also requiring a profile name.

Once the user completes onboarding (either by naming a profile or importing data), the app
proceeds to the Summary page and the onboarding screen is never shown again.

If the user dismisses the onboarding screen without completing it (e.g., closes the modal or
navigates away), the app silently creates a profile named **"Main"** as the primary profile and
proceeds normally. The onboarding screen is not shown again.

## Import / Export

Import and export are accessible from the app at any time via a button or menu item, but the
import action is intentionally more prominent on the first-run onboarding screen (see above)
than in normal operation, since first launch is the most common time a user would want to import.

**Export**
- Export all collection data (all profiles) as a JSON file in the canonical export format
- Ideally also export as XLSX; CSV is acceptable as a fallback. Spreadsheet export is a
  nice-to-have, not required.

**Import**
- Import a previously exported JSON file
- Ideally supports merging into existing data: when an imported profile name matches an existing
  profile, prompt the user to choose overwrite or skip
- Importing from spreadsheet formats is not required

Both the internal save format and the export format must include a format version field.
The app must be able to load and automatically migrate older format versions.

## Settings

A settings area (page or panel) for app-wide preferences. At minimum:

**Dark / Light mode**
- Manual toggle: dark or light
- Default: follow the system theme
- Preference is persisted across sessions

**Ignore unobtainable sets**
- Toggle: when enabled, sets where `Set::retirement_date()` returns a date in the past are
  excluded from the entire app — hidden from the Card Catalog, excluded from set completion
  counts, excluded from all probability calculations and query results
- Default: off (unobtainable sets appear in the catalog and completion, but their packs are
  excluded from recommendations by default — see §Summary Page)
- When this setting is on, the Obtainable filter on the Summary page is hidden entirely
- Persisted across sessions

**Ignore Premium Mission cards**
- Toggle: when enabled, cards where `CardVersion::source().name()` is `"Premium Mission"` are
  excluded from the entire app —
  hidden from the Card Catalog, excluded from set completion counts, excluded from probability
  calculations, and excluded from all query results
- Default: off (premium cards are shown)
- Persisted across sessions

**Ignore Gold Shop cards**
- Toggle: when enabled, cards where `CardVersion::source().name()` is `"Gold Shop"` are excluded
  from the entire app in the
  same way as above
- Default: off
- Persisted across sessions

These two settings are independent. Users may ignore one premium source but not the other.

**Merge duplicate printings**
- Toggle: when enabled, card versions that are physical reprints of each other (linked via
  `CardVersion::duplicates()`) are treated as a single logical card throughout the entire app:
  - The owned count for any version in a duplicate group is the **sum of owned counts across all
    versions in the group**, regardless of which sets those versions belong to
  - Completion percentages and denominators count the duplicate group as **one card**, not one
    per version
  - If multiple versions from the same duplicate group would otherwise appear in the same
    computation (e.g., both A1 025 and its duplicate in another set match the same filter), they
    are collapsed into a single entry using the combined count — never double-counted
- **Count editing with this setting enabled**: individual card version entries remain separately
  accessible in the catalog and card details — duplicates are not collapsed in the UI. The
  displayed count for any version is the merged sum, but editing always modifies that specific
  version's underlying count in storage. The decrement button is disabled (no-op) when the
  individual version's count is 0, even if the merged sum is still > 0 — this prevents the
  underlying stored count from going negative.
- Default: off (each card version is counted independently)
- Persisted across sessions
